對象：前端工程師（不熟悉後端） 版本：v1.0 日期：2026-05-17

0. 先建立心智模型

在開始學任何技術名詞之前，先想清楚我們要解決的問題。

0.1 為什麼純前端不夠用

你目前的劇本對練平台是純前端：劇本與錄音都存在瀏覽器的 IndexedDB 裡。這帶來三個無解的問題：

  1. 換裝置就消失。手機錄的音檔，電腦看不到。
  2. 清瀏覽器資料就消失。使用者點一下「清除快取」，半年的錄音就沒了。
  3. 無法分享。導演想看你錄的版本？沒辦法傳給他。

要解決這三個問題，資料必須存在「某個雙方都能連到的地方」。這個地方就是「雲端」，提供這個地方的程式就是「後端」。

0.2 心智模型：把「後端」想成一個共享硬碟

┌────────────┐ ┌─────────┐ ┌────────────┐ │ 你的手機 │ ──────► │ 共享硬碟│ ◄────── │ 你的電腦 │ └────────────┘ │ （雲端）│ └────────────┘ └─────────┘ ▲ │ ┌────────────┐ │ 導演的電腦│ └────────────┘

但這個共享硬碟有幾個前端硬碟沒有的特性：

  • 它要能判斷誰可以看什麼（你不能看別人的錄音 → 授權）
  • 它要能確認你真的是你（不能任何人都自稱是你 → 認證）
  • 它要能同時處理很多人（不能你寫到一半另一個人也寫，資料就亂了）
  • 它要能分類儲存（小東西如劇本文字 vs 大東西如音檔，存的方式不一樣）

本文件接下來的每一章，都是在解釋「這個共享硬碟」內部的一個機制。

前端類比

「後端」對前端工程師來說，就是一個你不能直接讀寫的 source of truth。你只能透過 HTTP request 跟它對話。它的角色等於：把所有人的 localStorage 集中到一台機器，加上「誰可以看誰的」規則。

1. 後端到底是什麼

1.1 後端 = 「在伺服器上跑的程式」

「後端」這個詞涵蓋很廣，但對我們這個專案來說，它指的是三類東西的組合：

類別用途本專案的選擇
資料庫（Database）存結構化資料（劇本內容、使用者資料、權限關係）Supabase 內建的 Postgres
物件儲存（Object Storage）存「檔案」這種大型二進位資料（音檔 Blob）Cloudflare R2
運算層（Compute）跑一些前端不能跑的程式（驗證、簽發 URL、cleanup）Supabase Edge Functions

1.2 資料庫 vs 物件儲存：為什麼要分開

資料庫（Postgres）

  • 適合小而多的結構化資料
  • 有 schema（欄位、型別）
  • 可以用 SQL 查詢、JOIN、聚合
  • 單筆建議 < 1 MB
  • 例：使用者列表、劇本台詞文字

物件儲存（R2）

  • 適合大而少的二進位檔案
  • 沒有 schema，就是「一個 key 對一個檔」
  • 不能做查詢，只能「取出整個檔」
  • 單筆可以到 GB 等級
  • 例：音檔、影片、圖片、PDF
注意

常見錯誤是把音檔直接存進資料庫的 bytea 欄位。技術上做得到，但會帶來：(1) DB 備份檔暴增、(2) 查詢 row 時連 Blob 一起拉、(3) 大幅吃光 free tier 配額。大檔一律走物件儲存。

1.3 為什麼前端不能直接連資料庫

新手常問：「既然要存資料庫，前端 JS 直接 connect('postgres://...') 不就好了嗎？」答案是絕對不行，三個理由：

  1. 密碼會被偷。前端代碼是公開的，任何人按 F12 都能看到。資料庫密碼放進前端等於把銀行金庫密碼貼在門口。
  2. 資料庫協定不是給瀏覽器用的。Postgres 用的是 TCP socket，瀏覽器只支援 HTTP / WebSocket，技術上根本連不上。
  3. 權限沒辦法做。就算連得上，也代表前端能讀寫整個資料庫，沒有任何隔離。

所以前端與資料庫之間，永遠要有一層「會驗證身份、會檢查權限」的中介。這層可以是：

  • 傳統作法：自己寫一個 Node.js / Go / Python API server
  • 新派作法（我們選的）：用 Supabase — 它幫你寫好了這層，前端透過 supabase-js 用 HTTP 與它對話
前端類比

Supabase 的角色像一個「幫你包好的 BaaS」（Backend as a Service）。你不需要寫 Express、不需要設定資料庫連線池、不需要自己寫 JWT 驗證 — 都包好了。你只需要呼叫類似 supabase.from('scripts').select('*') 這種 API。

2. 認識 Supabase

2.1 一句話定義

Supabase 是一個開源的 BaaS（Backend as a Service），底層是 PostgreSQL，把資料庫、認證、檔案儲存、即時推播打包成可直接用 HTTP 呼叫的服務。

網路上常拿來與 Firebase 對比：

Firebase（Google）Supabase
底層 DBFirestore（NoSQL 文件式）PostgreSQL（關聯式）
查詢語言自家 APISQL（標準）
開源否是
可自架否是
權限模型Security Rules（DSL）Row Level Security（PostgreSQL 內建）

2.2 Supabase 包含的子服務

子服務本專案用到用途
Database (Postgres)✓存劇本、使用者、segment metadata
Auth✓使用者註冊登入、JWT 發放
Storage✗檔案儲存（free 只有 1 GB，所以音檔不放這）
Edge Functions✓跑後端程式碼（簽 R2 presigned URL）
Realtime✓訂閱 DB 變動，做跨裝置即時推播

2.3 你的前端如何與 Supabase 對話

透過官方 SDK @supabase/supabase-js。建立 client 後，操作 DB 看起來像這樣：

// 在前端瀏覽器執行
import { createClient } from '@supabase/supabase-js';

const supabase = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL,
  process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY
);

// 讀取資料 — 這背後其實是發 HTTP GET 給 Supabase
const { data, error } = await supabase
  .from('scripts')
  .select('*')
  .eq('owner_id', userId);
關鍵理解

supabase.from('scripts').select('*') 並不是「直接連 Postgres」。它在底層是發一個 HTTP request 到 Supabase 的 REST API（叫做 PostgREST），由 PostgREST 把你的 query 翻譯成 SQL 後執行，再把結果 JSON 回傳。

所以你寫的 JS 看起來像在操作資料庫，但實際上每個動作都是一個 HTTP 來回。

2.4 Free Tier 配額（2026）

項目限制本專案會碰到嗎
Database 大小500 MB不會（劇本文字 + metadata 遠低於此）
Monthly Active Users50,000不會（< 50 人團隊）

3. 認識 Cloudflare R2

3.1 一句話定義

Cloudflare R2 是一個 S3 相容的物件儲存服務，最大特色是「下載永遠免費」。

3.2 什麼是 S3？什麼是物件儲存？

「S3」是 AWS 的物件儲存服務（Simple Storage Service），業界事實標準。「S3 相容」意思是別家服務（R2、MinIO、Backblaze B2）都實作了一樣的 API，你可以用同一套 SDK（@aws-sdk/client-s3）操作。

物件儲存的概念非常簡單：

┌───────────────────────────────┐ │ Bucket（桶） │ │ │ │ key value │ │ ────────────────── ────── │ │ user/avatar.jpg [Blob] │ │ audio/scene1.webm [Blob] │ │ audio/scene2.webm [Blob] │ │ │ └───────────────────────────────┘

就是一個巨大的 key-value 倉庫。key 是字串（通常長得像檔案路徑），value 是檔案。沒有資料夾的真實概念，只是用 / 在 key 裡組織而已。

3.3 R2 vs Supabase Storage vs AWS S3

R2Supabase StorageAWS S3
Free 儲存10 GB1 GB5 GB（12 個月）
Egress 費用永遠 $0計入頻寬$0.09 / GB
儲存費用（超過 free）$0.015 / GB / 月$0.021 / GB / 月$0.023 / GB / 月
S3 相容是否（自家 API）—

3.4 為什麼「Egress 永遠 $0」這麼重要

Egress = 「資料流出雲端」的頻寬。每次使用者播放音檔，那個 Blob 就要從雲端「流」到使用者的瀏覽器。傳統雲端服務會為這段流量收錢。

情境：5 個團員，每人練習一段 10 MB 的劇本，每天聽 20 次 => 每人每天 200 MB egress => 全團每月 5 人 × 200 MB × 30 天 = 30 GB egress 在 AWS S3：$0.09 × 30 = $2.7 / 月（看起來不多） 在使用者增加到 50 人時：$27 / 月 在熱門劇本被回放 100 次時：$270 / 月（爆炸） 在 R2：$0（無論多少都是零）
關鍵設計

R2 的「零 egress」是本專案能維持近乎零成本的單一最重要決策。其他選擇只要使用情境一變熱門，帳單就會失控。

3.5 R2 怎麼用？

R2 本身不認 JWT、不認使用者，它只認「Access Key + Secret」（像帳號密碼）。所以前端不能直接拿著 access key 去操作 R2 — 那等於把整個 bucket 的金庫鑰匙塞給每個瀏覽器。

解法是「Presigned URL」：由你的後端（Edge Function）用 access key 簽出一個短期、限定動作、限定物件的 URL，把這個 URL 給前端，前端用這個 URL 直接與 R2 對話。詳見第 7 章。

4. 認證（Authentication）

4.1 認證 vs 授權 — 兩個常被搞混的詞

認證 Authentication授權 Authorization
問題你是誰？你能做什麼？
例子登入時輸入 email 與 magic link登入後能不能讀別人的劇本
常見技術密碼、Magic Link、OAuth、JWTRBAC、ABAC、Row Level Security

本章只談認證；第 5 章談授權。

4.2 我們選的方案：Magic Link

Magic Link 是一種「無密碼認證」。流程是：

  1. 使用者在登入頁輸入 email
  2. Supabase 寄一封信，內含一個一次性連結（含 token）
  3. 使用者點連結 → 瀏覽器把 token 交給 Supabase → 換到 JWT → 登入完成
為什麼選 Magic Link

(1) 不用做密碼欄位、不用做忘記密碼流程。(2) 比密碼安全 — 沒有密碼可以洩漏。(3) 小團隊內部用，email 寄達率不是問題。後續若要擴可以再加 Google OAuth。

4.3 JWT 是什麼

JWT（JSON Web Token）是一段「由伺服器簽過名的 JSON」，用來證明「你已經登入、你是誰」。長得像這樣：

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJhYmMxMjMiLCJlbWFpbCI6ImpqQGcuY29tIn0.x7VsR8Tk...

三段以 . 分隔，每段是 base64 編碼：

段內容誰寫的
1. Header{ "alg": "HS256", "typ": "JWT" }伺服器
2. Payload{ "sub": "abc123", "email": "..." }伺服器
3. Signature用伺服器秘鑰對 Header+Payload 算 HMAC伺服器
注意

JWT 是 base64 編碼，不是加密。任何人複製貼到 jwt.io 都能看到內容。所以：(1) 別在 payload 放敏感資訊（密碼、信用卡）。(2) 安全性來自「沒有伺服器的秘鑰，就無法偽造簽名」— 改 payload 一個字，signature 就對不上，後端會拒收。

4.5 JWT 怎麼被帶回伺服器

登入後，supabase-js 會把 JWT 存起來。每次你呼叫 supabase.from(...)，它會自動把 JWT 放進 HTTP request 的 Authorization: Bearer <jwt> header，後端就知道是誰在發 request。

儲存位置依設定不同：

HttpOnly CookieJS 讀不到、防 XSS 偷 token需要 SSR 才能用，要小心 CSRF

5. 授權與 Row Level Security (RLS)

5.1 問題定義

登入後，你已經知道使用者「是誰」（認證完成）。下一個問題是：他能讀寫哪些資料？

舉例：

  • 使用者 A 不能讀使用者 B 的劇本
  • 使用者 A 可以讀 B 共享給他的劇本（但不能編輯）
  • 使用者 A 只能修改自己錄的音檔，不能改 B 的

5.2 傳統作法（前端工程師常想到的）

❌ 在前端 if-else 判斷

if (script.ownerId !== user.id) {
  throw new Error('no permission');
}
// 然後才呼叫 supabase 讀資料

問題：任何人改前端 JS，就能繞過。前端的 if 永遠是「UX 提示」，不是「安全」。

❌ 在 API server 寫 middleware

app.get('/api/scripts/:id', (req, res) => {
  if (req.user.id !== script.ownerId) {
    return res.sendStatus(403);
  }
  ...
});

問題：能用但麻煩 — 每個 endpoint 都要記得寫；忘了一個就是漏洞。

5.3 我們的作法：Row Level Security（RLS）

RLS 是 PostgreSQL 內建的功能：在資料庫層為每個 table 寫「哪些 row 對哪些使用者可見」的規則。一旦啟用，所有 query 都會自動套用這個過濾條件，不管是前端來的 query、後端來的 query、誰寫的 query 都跑不掉。

規則看起來像這樣（SQL）：

-- 啟用 RLS
alter table scripts enable row level security;

-- 政策：只有 owner 自己才能看到自己的劇本
create policy "own scripts"
  on scripts for select
  using (owner_id = auth.uid());

-- 政策：被 share 的人也能看
create policy "shared scripts"
  on scripts for select
  using (
    exists (
      select 1 from script_shares
      where script_shares.script_id = scripts.id
        and script_shares.user_id = auth.uid()
    )
  );

auth.uid() 是 Supabase 提供的 SQL 函式，它會從 JWT 的 sub 欄位取出當前使用者 ID。

5.4 RLS 在執行時做什麼

前端發了一個 query：

supabase.from('scripts').select('*');

PostgREST 翻成 SQL：

select * from scripts;

但 PostgreSQL 因為有 RLS，自動加上 WHERE：

select * from scripts
where  owner_id = auth.uid()                            -- own scripts policy
   or  exists (select 1 from script_shares ...);    -- shared scripts policy
為什麼這個設計優雅

(1) 規則寫在 DB 層，沒有任何 API 路徑能繞過。(2) 即便前端 bug 寫錯 query，最壞情況也只是「拿不到資料」，不會洩漏資料。(3) 新增 endpoint 不用重寫權限檢查，policy 自動生效。

5.5 RLS 的「狀態」很重要

危險

RLS 預設是「關閉」的。如果你建了 table 但忘了 alter table ... enable row level security，那個 table 對任何持有 anon key 的人都是完全公開的 — 任何人都能讀寫所有 row。

建表後第一件事必須是啟用 RLS、寫 policy；本專案的 SPEC 已明文要求 M30~M34 每個 table 都要做這件事。

6. API 是什麼

6.1 對前端工程師，API 就是 fetch 在打的東西

你已經很熟 API 了 — 每次寫 fetch('/api/something') 就是在打 API。本章想釐清的是：本專案有哪些 API、誰提供、長什麼樣。

6.2 本專案會用到的三類 API

類型誰提供本專案怎麼用
PostgREST（自動 DB API）Supabase 自動生成讀寫 scripts / audio_segments / practice_states
Auth APISupabase 內建登入、登出、refresh token
Edge Function（自定義）我們自己寫，跑在 Supabase簽 R2 presigned URL、清理孤兒物件

6.3 PostgREST：把 table 變成 REST API

Supabase 內建一個叫 PostgREST 的工具，把每個 Postgres table 自動變成 REST endpoint：

// 你寫的 JS
supabase.from('scripts').select('*').eq('id', 'abc');

// 底層發出的 HTTP request
GET https://xxx.supabase.co/rest/v1/scripts?id=eq.abc
Authorization: Bearer <jwt>
apikey: <anon-key>

RLS 在這層生效。Supabase 收到 request → 把 JWT 解出 user_id → 把 query 丟給 Postgres → Postgres 套用 RLS policy → 回傳結果。

6.4 Edge Function：你自己寫的後端程式

有些事 PostgREST 做不到 — 例如「幫前端簽一個 R2 presigned URL」（需要用到 R2 secret key，這 key 絕不能進前端）。這時就需要寫 Edge Function。

Edge Function 是一段你寫的 Deno（TypeScript）程式，部署到 Supabase 後變成一個 HTTP endpoint：

// supabase/functions/audio-upload-url/index.ts
import { serve } from "https://deno.land/std/http/server.ts";

serve(async (req) => {
  // 1. 驗 JWT
  const jwt = req.headers.get('Authorization')?.replace('Bearer ', '');
  const { data: user } = await supabase.auth.getUser(jwt);

  // 2. 用 R2 secret 簽一個 5 分鐘 PUT URL
  const uploadUrl = await signR2PutUrl(...);

  return new Response(JSON.stringify({ uploadUrl }));
});

前端透過 supabase.functions.invoke('audio-upload-url', {...}) 呼叫它。

前端類比

Edge Function 就像 Next.js 的 API Route 或 Vercel Function — 一段你寫的、跑在伺服器的 TypeScript。差別是它跑在 Supabase 的 Deno runtime，能直接用 Supabase 的 service-role key 操作 DB（繞過 RLS）。

7. Presigned URL 直傳模式

這是整個架構最關鍵、最容易誤解的一章。請仔細看。

7.1 問題：音檔要怎麼上傳到 R2？

有兩個直覺但都錯的選項：

❌ 選項 A：前端拿 R2 secret 直傳

把 R2 access key + secret 寫在前端 → 用 @aws-sdk/client-s3 直接 PUT。

後果：任何人看你的 JS bundle 就能拿到 secret，整個 bucket 就是他的了。

❌ 選項 B：音檔流經 Edge Function

前端 POST Blob 給 Edge Function → Edge Function 再轉寫到 R2。

後果：Edge Function 有 CPU 時間限制（free tier 通常 50ms）；大檔超時必失敗。即便沒超時，這條路徑也吃 Supabase egress 流量。

7.2 正確解法：Presigned URL

Presigned URL 是一個「事先由後端簽好名、限時、限定動作」的 URL。前端拿著這個 URL 直接與 R2 對話，不需要任何 secret。

它是怎麼運作的

R2（與 S3）支援這種機制：用 access key + secret 對「這個動作 + 這個物件 key + 這個過期時間」做一個 HMAC-SHA256 簽名，把簽名塞進 URL 的 query string。當有人用這個 URL 來 PUT 時，R2 會：

  1. 檢查 URL 是否過期（query 裡的 expiry 時間）
  2. 用 access key 重新算一次簽名，看是否與 URL 裡的一致
  3. 檢查動作是否一致（只能做被簽的動作 — PUT 就不能 GET）
  4. 檢查物件 key 是否一致（不能拿這個 URL 去寫別的 key）

四項都對才允許。否則 403。

實際 URL 長相

https://bucket.r2.cloudflarestorage.com/scripts/abc/123/12.webm
  ?X-Amz-Algorithm=AWS4-HMAC-SHA256
  &X-Amz-Credential=...
  &X-Amz-Date=20260517T120000Z
  &X-Amz-Expires=300                // 5 分鐘 = 300 秒
  &X-Amz-SignedHeaders=host
  &X-Amz-Signature=<hmac-sha256-hex>

為什麼這個設計很重要

音檔的 PUT 與 GET 流量都不經過 Edge Function，是前端與 R2 直連。Edge Function 只負責「簽 URL」這件事（幾 KB 資料）。所以：

  • Edge Function 的 CPU 時間配額永遠燒不完
  • R2 的 egress 永遠 $0
  • 整個架構在 50 GB 音檔量級下，月成本 < $1

7.5 為什麼 URL 要設過期時間

如果 URL 永久有效，被 leak 出來（例如轉貼到群組、被 referer 帶到外站、被瀏覽器歷史保存）就等於這個物件公開了。5 分鐘 TTL 把風險窗口縮到極小，過期後 URL 自動失效，要再用就重簽。

8. 資料同步協定

8.1 目標

  • 跨裝置同步：手機改了東西，電腦看得到
  • 離線可用：沒網路也能練（已快取的劇本與音檔）
  • 衝突處理：兩台裝置同時改同一份劇本，怎麼合併

8.2 架構：Cloud 為真相，IDB 為快取

┌───────────────────────────┐ │ Source of Truth │ │ ─────────────── │ │ Supabase Postgres │ ← 權威資料在這裡 │ Cloudflare R2 │ └─────────────▲─────────────┘ │ │ 同步 ▼ ┌───────────────────────────┐ │ 本地 IndexedDB │ ← 離線時的「副本」 │ ───────────────── │ │ scripts, audio segments, │ │ practice_states │ └───────────────────────────┘

8.3 寫入路徑：樂觀更新 + 背景同步

當使用者改了東西，不要等雲端回應再更新 UI — 那樣會卡。改成：

  1. 立即寫 IDB → React state 更新 → UI 即時反映
  2. 同時發 request 寫雲端
  3. 雲端成功 → 標記這筆 record 為「已同步」
  4. 雲端失敗（離線 / 500）→ 標記為「dirty」，存進 sync_queue
  5. online 事件觸發時，依序 flush sync_queue
前端類比

這就是 React Query / SWR 的 onMutate + onError 的概念。本質是 optimistic update：先假裝成功更新 UI，背景失敗再回滾或重試。

8.4 讀取路徑：IDB 優先，背景對齊

  1. 頁面 mount → 從 IDB 讀劇本 → 立刻渲染（毫秒級）
  2. 同時發雲端 query → 拿到結果 → 比較 updated_at → 較新者覆蓋 IDB → 觸發 React 重新渲染
  3. 同時訂閱 Supabase Realtime → 雲端有人改 → patch IDB → 觸發重新渲染

8.5 衝突解決：Last Write Wins（LWW）

規則：哪邊的 updated_at 比較新，就以哪邊為準。

場景：

  • A 裝置 10:00 改劇本
  • B 裝置 10:05 改同一份劇本（A 的修改還沒同步到雲端）
  • 網路恢復後 B 的修改先到，雲端 updated_at = 10:05
  • A 嘗試 push 自己的 10:00 修改 → 後端比較發現自己的 updated_at 較舊 → 拒絕，A 拉回雲端版本覆蓋本地
注意

LWW 的缺點是 會丟資料：A 在 10:00~10:05 間做的修改，會被 B 的版本覆蓋掉。對小團隊、低同時編輯機率的場景可接受；要徹底解決需要 CRDT（如 Yjs、Automerge），複雜度高一個數量級，本專案 v1 不做。

8.6 為什麼不做 CRDT

CRDT（Conflict-free Replicated Data Type）是一種資料結構，能讓多端同時修改後自動 merge 而不丟資料。Google Docs、Linear、Figma 都用類似技術。

但它代價很大：

  • 每次修改都要產生 operation log，資料量翻倍以上
  • 需要重寫整個資料模型（不能用普通 row，要用 op log）
  • 複雜度進階很多，debug 困難

小團隊（< 50 人）+ 個人練稿 + 偶爾共享的場景，LWW 已經夠用。等真的有「兩個人同時編輯同一個劇本」的痛點，再上 CRDT 不遲。

9. 環境變數與安全

這章是新手最容易出大事的地方。請完整看完。

9.1 NEXT_PUBLIC_ 前綴的真正意義

Next.js 規定：環境變數以 NEXT_PUBLIC_ 開頭的，會在 build 時被「內嵌」進前端 bundle，瀏覽器拿得到。沒這個前綴的，只有 server 端（API Route、Server Component、Edge Function）能讀。

.env.local ┌─────────────────────────────────────────────────┐ │ NEXT_PUBLIC_SUPABASE_URL=https://xxx.co │ ← 會被打包進前端 JS │ NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJh... │ ← 會被打包進前端 JS │ SUPABASE_SERVICE_ROLE_KEY=eyJh... │ ← 不會打包，server 才讀得到 │ R2_SECRET_ACCESS_KEY=abc123... │ ← 不會打包，server 才讀得到 └─────────────────────────────────────────────────┘ ↓ next build client.js（瀏覽器下載） ┌─────────────────────────────────────────────────┐ │ const url = "https://xxx.co"; │ │ const key = "eyJh..."; ← anon key 直接內嵌 │ │ // service-role 與 R2 secret 完全不在此檔內 │ └─────────────────────────────────────────────────┘
關鍵理解

NEXT_PUBLIC_* 變數不是「設定」，是「公開常數」。寫進去就等於貼到公告欄上。

9.2 三把 key 的角色（很重要）

Key權限放哪洩漏會怎樣
ANON_KEY 受 RLS 約束的公開 key 前端 + 公開 repo OK 沒事 因為 RLS 擋著
SERVICE_ROLE_KEY 繞過所有 RLS 的管理員 key 只放 Edge Function Secrets 災難 整個 DB 任由他人讀寫
R2_SECRET_ACCESS_KEY 整個 R2 bucket 的管理員 key 只放 Edge Function Secrets 災難 整個 bucket 任由他人讀寫刪
危險：絕對不可以做的事
  • 把 SERVICE_ROLE_KEY 變數名稱寫成 NEXT_PUBLIC_SERVICE_ROLE_KEY → 整個 DB 公開
  • 把 R2_SECRET_ACCESS_KEY 寫成 NEXT_PUBLIC_R2_SECRET_ACCESS_KEY → 整個 bucket 公開
  • 把 .env.local commit 進 git → 即使 push 到 private repo，也算外洩（協作者、CI、log 都可能看到）
  • 在 Next.js Server Component 內寫 console.log(process.env.SERVICE_ROLE_KEY) → 在某些 streaming 場景可能流到前端 HTML

萬一不小心 commit 了 secret：立刻去 Supabase / R2 dashboard 重新生成一組新的 key，舊的廢掉。從 git 移除是不夠的 — 已經暴露了就當作已外洩。

9.3 為什麼 ANON_KEY 是「安全的公開」

新手會誤以為 anon key 應該保密。其實它設計上就是公開的 — 它本身沒有任何權限，只是一張「我是匿名訪客」的入場券。實際能做什麼，完全由 RLS policy 決定。

類比：anon key 像 API 的 base URL，service-role key 才像密碼。

11. 常見誤解與陷阱

11.1 「我在前端檢查了權限應該就安全了」

不安全。前端代碼公開，按 F12 就能改。所有安全檢查都必須在後端（RLS / Edge Function）做。前端只負責 UX 提示（例如灰掉按鈕）。

11.2 「JWT 有加密，內容不會被看到」

錯。JWT 只是 base64 編碼，任何人都能解出 payload。安全性來自「沒有秘鑰無法偽造簽名」。所以 payload 不可放敏感資訊。

11.3 「我用 anon key 就好，service-role 那個 dashboard 給的 key 我先不用」

正確。本專案前端永遠只用 anon key。service-role 只在 Edge Function 的特定場景用（例如要繞過 RLS 做 admin 任務）。

11.4 「Edge Function 不就是 API Route 嗎？我用 Next.js API Route 寫不行嗎？」

技術上可以。但用 Supabase Edge Function 的好處：(1) 與 Supabase 同一個機房，DB latency 低；(2) 用 Supabase 內建的 service-role / auth helper 比較方便；(3) Vercel API Route 有冷啟動，Supabase Edge Function 預熱速度比較穩。第一版建議放 Supabase。

11.5 「我把音檔的 R2 物件 key 設成公開，就不用 presigned URL 了」

不行。一旦設成公開，任何人猜到 key（key 通常是可預測的格式）就能下載。音檔含真人聲音，屬於個資，必須走 presigned URL。

11.6 「presigned URL 有過期時間，那播放音檔到一半過期會中斷嗎？」

不會。過期檢查只在 R2 收到 request 時做一次。連線建立後即便 URL 過期，那個串流仍會正常完成。但下次重新請求就要新 URL。

11.7 「我的 supabase-js 在 Next.js Server Component 用會出問題嗎？」

會。@supabase/supabase-js 是給瀏覽器用的，不會處理 cookies。要在 SSR / Server Component 用必須改用 @supabase/ssr 套件，它會正確處理 cookies 讀寫。本專案 M30 會接這個。

11.8 「資料庫 schema 改了怎麼辦？要不要寫 migration？」

要。Supabase 支援用 SQL 檔做 migration（放在 supabase/migrations/）。每次改 schema 都要寫一個 migration 檔，並 commit 進 git，這樣其他人 / CI 才能重現你的 DB 狀態。本專案 M30 起會建立這套流程。

11.9 「使用者刪帳號了，他的音檔會自動消失嗎？」

Postgres 的 row 會（我們設定了 on delete cascade），但 R2 物件不會自動消失。R2 不知道 Postgres 在做什麼。本專案 M35 會加一個每日 cron Edge Function，掃出「Postgres 沒有對應 row 的 R2 物件」並刪除。

12. 詞彙對照表

所有本文件出現的後端名詞，配上前端類比與一句話定義。

BaaS（Backend as a Service）
把「後端常用功能（DB、Auth、Storage）」整包提供的雲端服務。Supabase、Firebase、AWS Amplify 都是。前端類比：像 Tailwind CSS 一樣，是「不用從零寫」的後端版本。
Postgres / PostgreSQL
開源關聯式資料庫；用 SQL 操作；支援 JSON、RLS、Full-text search 等先進功能。Supabase 底層就是它。
PostgREST
把 Postgres table 自動變成 REST API 的工具。你寫 supabase.from('scripts').select() 背後就是它在工作。
RLS（Row Level Security）
Postgres 內建功能，可以在 row 層為每個使用者設定可見/可寫條件。前端類比：像 TypeScript 的 type narrowing — 規則寫在某處，全程式自動套用。
Policy
RLS 的具體規則。每個 table 可以有多個 policy（OR 關係）。
JWT（JSON Web Token）
一段「伺服器簽過名的 JSON」，用來證明你已登入。可被任何人讀，但無法被偽造（除非有秘鑰）。
Magic Link
無密碼登入方式：輸入 email → 收到一封含一次性 token 的信 → 點連結即登入。
Authentication（認證）
確認「你是誰」。實作通常是 JWT / Session。
Authorization（授權）
確認「你能做什麼」。實作通常是 RLS / RBAC。
Object Storage（物件儲存）
專門存大檔的服務。沒有 schema，就是 key-value 倉庫。例：R2、S3。
S3-compatible
實作了 AWS S3 API 的服務，可以用同一套 SDK 操作。R2、MinIO、Backblaze B2 都是。
Bucket
物件儲存的「桶」，類似一個 namespace。你的物件 key 在這個 bucket 內唯一。
Presigned URL
由後端用 secret 簽過的、限時、限定動作的 URL。前端拿著它直接讀寫物件儲存，不需要 secret。
Egress
「資料流出雲端」的頻寬。多數雲端服務會收費，R2 不收（這是它最大賣點）。
Edge Function
跑在雲端邊緣節點的小段程式碼，接收 HTTP request 後執行。可以做 PostgREST 做不到的事（簽 URL、複雜計算）。
Realtime（postgres_changes）
Supabase 提供的 WebSocket 服務。前端訂閱某個 table 的變動，雲端有 INSERT/UPDATE/DELETE 就推到前端。用來做跨裝置即時同步。
Service Role Key
Supabase 的管理員 key，繞過所有 RLS。絕對不能放前端，只能在 Edge Function 用。
Anon Key
Supabase 的匿名訪客 key，受 RLS 約束，設計上就是公開的。放前端沒事。
LWW（Last Write Wins）
衝突解決策略：哪邊的更新時間較晚就以哪邊為準。簡單但會丟資料。
CRDT（Conflict-free Replicated Data Type）
能自動合併多端修改的資料結構，不丟資料。複雜度高，本專案 v1 不做。
Optimistic Update（樂觀更新）
UI 不等後端回應就先更新；背景再同步。失敗時回滾。React Query / SWR 內建這套。
migration（資料庫遷移）
一段描述「DB schema 怎麼變」的 SQL，commit 進 git 後人人都能重現相同 schema。與 IndexedDB 的 migration 觀念相同。
Cascade（連動刪除）
Postgres 的 on delete cascade 設定：父 row 刪除時，所有引用它的 child row 也自動刪除。
